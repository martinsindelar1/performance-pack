Internal Setup Guide

Build a self-updating dashboard
with a custom Claude skill + GitHub Pages

A start-to-finish walkthrough of how the Complex Performance Pack publishes itself: a local Claude Code skill that builds the HTML report, versions every run, and auto-deploys a shareable public link. Follow this to set up the same thing for your own report.

⏱️ ~30 min first-time setup 💻 macOS · zsh · git 🤖 Claude Code + a skill bundle 🌐 GitHub Pages hosting (free)

1

Prerequisites & accounts

What you need before touching the terminal.

  • A GitHub account. The repo can be private; only the published site is public.
  • Git installed. Check with git --version. On macOS it ships with the Xcode command-line tools.
  • Claude Code running in your terminal.
  • The skill bundle — a .skill file (it's just a zip). You can export it from Claude, or copy the .claude/skills/ folder from an existing setup.
  • Your data exports ready when you want to run it (Triple Whale Pins .xlsx, Shopify .csv, Klaviyo, etc.).

Confirm your git identity is set so commits are attributed to you:

terminal# check who git thinks you are
git config --global user.name
git config --global user.email

# set them if blank
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
2

Initialize the git repo

Turn your project folder into a tracked repository.

From inside your project folder (the one holding your dashboard HTML):

terminal# start git in the current folder
git init

# GitHub Pages serves a file named index.html as the homepage,
# so make your dashboard the index
cp your_dashboard.html index.html

git add -A
git commit -m "Initial dashboard"
Why index.html?GitHub Pages automatically serves index.html at the root URL. Keeping your live dashboard named index.html means the shareable link just works with no extra path.
3

Install & authenticate the GitHub CLI

The gh tool lets Claude create the repo and push without leaving the terminal.

If you have Homebrew: brew install gh. No Homebrew? Grab the official binary directly (Apple Silicon shown — use amd64 on Intel):

terminal · no-homebrew installmkdir -p ~/bin # a folder for personal binaries
cd /tmp
curl -sLO https://github.com/cli/cli/releases/download/v2.94.0/gh_2.94.0_macOS_arm64.zip
unzip -oq gh_2.94.0_macOS_arm64.zip
cp gh_2.94.0_macOS_arm64/bin/gh ~/bin/gh
chmod +x ~/bin/gh
~/bin/gh --version  # verify

Then log in. If you already authenticate git to GitHub over HTTPS, your token is in the macOS keychain and you can reuse it:

terminal · auth# reuse the token already stored for github.com
security find-internet-password -s github.com -w | ~/bin/gh auth login --with-token

# …or just run the interactive flow and follow the browser prompts
~/bin/gh auth login

~/bin/gh auth status  # confirm you're logged in
PATH tipIf you installed to ~/bin, add it to your PATH so you can type gh anywhere: append export PATH="$HOME/bin:$PATH" to your ~/.zshrc, then open a new terminal.
4

Create a private repo & push

One command creates the GitHub repo, links it, and pushes your first commit.

terminalgh repo create your-repo-name \
  --private \
  --description "My performance dashboard" \
  --source . # use the current folder \
  --remote origin \
  --push    # push the existing commits right away
Private is fineYour code stays private. In the next step you publish only the rendered site. The repo itself never has to be public on a paid plan — though on a free plan, Pages requires the repo to be public (see next step).
5

Turn on GitHub Pages

This produces the public, shareable link. Done once in the browser.

  1. On a free account, first make the repo public: Settings → General → Danger Zone → Change visibility → Public. paid plans can keep it private
  2. Go to Settings → Pages (left sidebar).
  3. Under Build and deployment → Source, choose Deploy from a branch.
  4. Set Branch: main and folder / (root), then Save.
  5. Leave Enforce HTTPS checked.

After ~1 minute your dashboard is live at:

your public urlhttps://<your-username>.github.io/<your-repo-name>/
Heads up — it's publicAnyone with the link can view it, no login needed. The whole repo's history becomes public too if you made it public. Don't enable Pages until you're sure the data in the dashboard is safe to share.
6

Install the skill locally

A local project skill lives in your repo and overrides the bundled cloud version.

A .skill file is a zip. Unpack its contents into .claude/skills/ inside your repo:

terminalmkdir -p .claude/skills
unzip -oq your-skill.skill -d .claude/skills

# you should now see the skill folder with its SKILL.md
find .claude/skills -type f

The result is a normal folder — Claude discovers any SKILL.md under .claude/skills/:

.claude/skills/complex-performance-pack/
├── SKILL.md            # the instructions Claude follows
├── assets/
│   └── template.html   # canonical dashboard structure
└── references/         # data-extraction rules, edit map, copy patterns
    ├── data-extraction.md
    ├── edit-locations.md
    └── narrative-patterns.md
Local vs. bundledA project skill registers under its plain name (e.g. /complex-performance-pack). The original cloud skill stays namespaced (/anthropic-skills:complex-performance-pack). Typing the plain name runs your local copy — and a plugin update can't overwrite it.
Restart to load itClaude Code discovers skills at session start. After adding the folder, start a fresh session in this repo, then type / and confirm the skill appears in the list.
7

Wire up the auto-deploy workflow

Edit SKILL.md so each run archives a dated copy, updates the live file, and pushes.

The key is to give the skill's final step explicit, local instructions. Two important design choices:

  • Start from index.html, not the template. The editing is additive (append the new month to existing chart arrays), so basing each run on the current live file preserves all prior months. The template is only the fallback for a brand-new pack.
  • Date-stamp every run into artefacts/ so you keep a full history and can always roll back.

Add a closing step to SKILL.md along these lines:

.claude/skills/…/SKILL.md — final step### Archive, update index, deploy

# 1. keep a dated copy of every run
mkdir -p artefacts
cp _working.html "artefacts/performance-pack-$(date +%F).html"

# 2. update the file GitHub Pages serves
cp _working.html index.html
rm -f _working.html

# 3. publish — auto-push redeploys the public site
git add -A
git commit -m "Update performance pack (run $(date +%F))"
git push origin main
date +%FExpands to YYYY-MM-DD, giving filenames like artefacts/performance-pack-2026-06-11.html. Re-running on the same day simply overwrites that day's archive.
8

Track the skill, ignore the secrets

Keep local Claude settings private while version-controlling the skill itself.

By default people ignore the whole .claude/ folder — but that would also drop your skill. Ignore everything except the skills folder:

.gitignore# ignore local Claude settings, but keep the project skill tracked
.claude/*
!.claude/skills/
.DS_Store

# housekeeping
*.skill          # the zip bundle (already unpacked)
_working.html    # temp file the skill builds into

Commit the skill so it travels with the repo for your teammates:

terminalmkdir -p artefacts          # create the history folder
cp index.html "artefacts/performance-pack-$(date +%F).html"  # seed it

git add -A
git commit -m "Add local skill + versioned artefacts workflow"
git push origin main
End stateYour repo now holds the live dashboard, a dated history, and the skill — nothing else.
your-repo/
├── index.html                              # live site (GitHub Pages)
├── .gitignore
├── artefacts/
│   └── performance-pack-2026-06-11.html    # dated run history
└── .claude/skills/complex-performance-pack/ # the skill (tracked)
9

Run it every month

The whole point: one command turns new exports into a published update.

Open Claude Code inside the repo folder, then:

claude code/complex-performance-pack here's June's data

…and attach your files. You can also just say "update the performance pack, here's the new data" with files attached — the skill's description triggers it automatically. On each run it will:

  • Start from the current index.html (keeps every prior month)
  • Extract & cross-check the numbers from your exports
  • Add the new month across all charts, KPIs, and the written readout
  • Save a dated copy to artefacts/
  • Update index.html, commit, and push automatically → public site refreshes in ~1 min
Two rules① Always run it from the repo folder, or the local skill won't be active and the paths won't resolve. ② It auto-pushes to the public site — glance at your data before running.

Troubleshooting

  • Skill doesn't show under /. Restart Claude Code in the repo folder — skills load at session start.
  • It ran the cloud version instead. Use the plain name /complex-performance-pack; the namespaced /anthropic-skills:… is the old bundled one.
  • gh: command not found. Either add ~/bin to PATH, or call it by full path ~/bin/gh.
  • Push asks for a password. Run gh auth status; re-auth with gh auth login if needed.
  • Pages 404s. Confirm the repo is public (free plan), the branch is main, folder is / (root), and an index.html exists at the root. Give it a minute to build.
  • A month went missing. The skill must base off index.html, not the template. Restore from the matching file in artefacts/.
